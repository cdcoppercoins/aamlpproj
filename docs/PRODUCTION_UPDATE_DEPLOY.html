Routine guide — publish local changes to production (cPanel VPS)

Updated: May 24, 2026

Use this when: You have tested changes locally and want to update the live site at https://minilicenseplates.com.

Your setup (from project docs): Laravel app on cPanel · GitHub repo cdcoppercoins/aamlpproj · GitHub Desktop for commits · server app root typically /home/minilp/new.minilicenseplates.com/ · plate images not in Git (public/plates/ uploaded separately).

Never commit: .env, vendor/, public/plates/, or mlp_code/.

Before you start

  • Changes work on http://localhost:8000
  • You know whether this deploy includes new database migrations
  • You know whether this deploy includes new Composer packages (e.g. DomPDF)
  • You have cPanel login and (if available) SSH or Terminal access

Step-by-step

  1. Review what changed locally Open GitHub Desktop. Confirm the changed files look correct — especially public/main.css, new Blade views, controllers, migrations, and public/js/ if you added scripts. Do not stage .env or plate images.
  2. Commit on your PC In GitHub Desktop: write a short commit message describing the update → click Commit to main (or your working branch).
  3. Push to GitHub Click Push origin in GitHub Desktop. Wait until it completes with no errors.
  4. Pull on the production server Log into cPanel → Terminal (or use SSH). Run: 
    cd /home/minilp/new.minilicenseplates.com
git pull
    If you deploy by FileZilla instead of Git: upload only the changed files to the same path on the server, preserving folder structure.
  5. Install Composer packages (if needed) Run this when composer.json or composer.lock changed (e.g. DomPDF): 
    cd /home/minilp/new.minilicenseplates.com
composer install --no-dev --optimize-autoloader
    Skip this step if you only changed CSS, views, or PHP that uses existing packages.
  6. Run database migrations (if needed) Run this when new files exist in database/migrations/: 
    cd /home/minilp/new.minilicenseplates.com
php artisan migrate --force
    Skip if no new migrations. Never use phpMyAdmin alone for migrations — use Artisan.
  7. Storage link (one-time or if missing) Profile photos need the public storage symlink. Run once if not already done: 
    cd /home/minilp/new.minilicenseplates.com
php artisan storage:link
  8. Copy CSS to the live web root Production serves CSS from the document root. After editing public/main.css, copy it: 
    cp /home/minilp/new.minilicenseplates.com/public/main.css /home/minilp/public_html/main.css
    Adjust the public_html path if your live layout differs. Also upload any new files under public/js/, images, or banners if you added them.
  9. Clear Laravel caches 
    cd /home/minilp/new.minilicenseplates.com
php artisan config:clear
php artisan route:clear
php artisan view:clear
    If you see config errors, delete stale files in bootstrap/cache/ (keep .gitignore).
  10. Verify production .env (only when settings changed) In cPanel File Manager, open the server .env (not in Git). Typical values: 
    APP_URL=https://minilicenseplates.com
SESSION_DRIVER=database
DB_CONNECTION=mysql
DB_DATABASE=…
DB_USERNAME=…
DB_PASSWORD=…
    Do not overwrite production .env with your local copy.
  11. Test the live site Hard-refresh the browser (Ctrl+F5). Check:
    • Home page and new UI (banner rotator, styles)
    • Sign in / register / log out
    • My Collection and Edit by set
    • Gallery and Catalog Search
  12. Rollback plan (if something breaks)
    • Git: git log then git checkout <previous-commit> or revert the bad commit
    • Migrations: php artisan migrate:rollback --step=1 (only if a migration caused the problem)
    • Restore previous main.css from backup if styles broke
